## Git 運用ルール

### 基本方針

**コードを変更するたびに GitHub へプッシュする。**

具体的には以下の手順を毎回必ず実行すること：

1. 変更内容を確認する (`git diff`, `git status`)
2. 関連ファイルのみをステージングする (`git add <files>`)
3. 変更内容を端的に表すコミットメッセージを付けてコミットする
4. リモートへプッシュする (`git push`)

### コミットメッセージ規則

```
<type>: <概要（日本語可）>

例:
feat: タスクの並び替え機能を追加
fix: ドラッグ&ドロップ時のバグを修正
refactor: カード描画ロジックを整理
style: ボタンのスタイルを統一
docs: READMEを更新
```

### 禁止事項

- `git push --force` は原則禁止（main/master ブランチへの force push は絶対禁止）
- `--no-verify` によるフック回避は禁止
- `.env` や認証情報を含むファイルのコミットは禁止

### ブランチ運用

- `main` ブランチを本線とする
- 機能追加・修正は feature ブランチを切って作業し、完了後に main へマージする
- 単純な修正はそのまま main で作業してプッシュしても可

## 開発ガイドライン

- コメントは「なぜそうしているか (WHY)」が非自明な場合のみ書く
- エラーハンドリングはシステム境界（ユーザー入力・外部 API）にのみ追加する
- 必要以上の抽象化・将来の要件を見越した設計は行わない

---

## Claude Code への指示

- コードを変更した後は必ず GitHub へプッシュすること
- プッシュ前にユーザーへ確認を求めること（破壊的操作の場合は特に）
- セキュリティ上のリスクがあるコード（SQL インジェクション、XSS 等）は即座に修正する
